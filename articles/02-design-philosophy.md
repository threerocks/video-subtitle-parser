# Video Subtitle Parser 的设计哲学：字幕优先，ASR 兜底，证据留在本地

一个工具好不好用，不只看它能不能跑。

更要看它在失败时怎么做。

视频字幕解析这件事，失败路径很多。YouTube 有时有官方字幕，有时只有自动字幕，有时 timedtext 直接 429。B 站有的视频能拿到字幕，有的字幕为空，有的需要登录。抖音更像另一个世界，移动分享页、无水印地址、作者水印、平台水印，每一层都可能变。

如果只追求“跑通一次”，脚本很容易写。

但我想要的不是一次性脚本，而是一个可以被 AI 编程工具反复调用的素材入口。所以 Video Subtitle Parser 的第一条原则是：字幕优先，ASR 兜底。

能拿到字幕，就不要急着语音识别。

字幕通常更快，也更便宜。它可能来自作者，可能来自平台自动字幕，质量不一定完美，但至少有来源。只有当字幕不可用、为空、被限流，或者平台不给时，再下载音频走本地 `mlx-whisper`。

第二条原则是：所有结果必须落到本地。

这听起来像废话，但它决定了整个工具的形态。很多 AI 工作流会直接围绕 URL 运行。URL 很轻，复制方便，可它不是证据。它不能稳定 diff，不能保证明天还能访问，也不能让另一个模型接手时知道上一轮到底读了什么。

所以这个工具会写出固定的材料包。

清理后的全文，是写作入口。分段时间轴，是复查入口。metadata，是来源入口。视频和音频，是截图与复跑入口。

第三条原则是：项目术语不应该写死在工具里。

我最早用它处理《剑来》视频，所以会有很多术语修正：陈平安、宁姚、阮秀、骊珠洞天、落魄山。这些词对「一只国漫观察猫」很有用，但对别人没有意义。

开源版里，这些东西被挪到了 `--term-file` 和 `--initial-prompt`。

这一步很重要。通用工具不应该带着创作者自己的私货。你的项目有你的词表，我的项目有我的词表。工具只提供入口，不替任何项目预设世界观。

第四条原则是：视频帧不是天然可用的图片素材。

下载 MP4 很容易，截一张图也很容易。难的是判断这张图能不能用。平台 UI、字幕条、作者水印、画面右下角的标记，都可能让一张截图不适合作为公开配图。

所以 metadata 里会留下 watermark policy。它不是法律意见，也不是自动审查，只是提醒后续 AI 或人类编辑：视频帧要看过，不能因为文件在本地就直接发布。

这些原则组合起来，工具就变成了一件很具体的事：

把视频从“远程平台上的观看对象”，变成“本地项目里的可审计材料”。

这也是它能适配 Codex、Claude Code、Cursor、Windsurf、Aider 的原因。它不依赖某个 AI 工具的私有能力。任何 agent 只要能运行命令、读取文件，就能接入这套流程。

你可以让 Codex 解析一个 B 站人物志视频，生成《剑来》公众号文章。

也可以让 Claude Code 处理一个 YouTube 长访谈，提取观点和章节。

还可以让 Cursor 在项目里建立 `materials/` 目录，把每个视频都变成可复查的输入。

设计到最后，反而不复杂。

复杂的是平台。复杂的是内容。复杂的是人会不会在赶时间时把专题视频写成逐帧解析。

工具要做的，是把这些容易混在一起的东西拆开。先拿素材，再看类型，再写标题，再生成文章。每一步都有文件，每一步都能回看。

这就是 Video Subtitle Parser 的价值。

它不是一个“更聪明的总结器”。它是一个更稳的素材入口。
